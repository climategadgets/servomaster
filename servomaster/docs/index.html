Project Architecture
There are four basic architectural components in the project - Servo Controller, Servo, Coordinate Transformer, Transition Controller.
  • Servo Controller is a resource (servo) factory. It may also serve as a common behavior determinator, in cases when the hardware is not smart enough to define per-servo behavior. Also, this is an atomic entity for the resource discovery (for the protocols and hardware that support it).
  • Servo is a representation of either logical or physical device that is able to move between 0.0 and 1.0 coordinates. It is the terminal entity the end user should be concerned about.
  • Coordinate Transformer is obvious - it is necessitated by discrepancies between the logical outputs and physical device configurations. Typical example is rotational to linear transfomation.
  • Transition Controller is necessary when the servo behavior must be smarter than just "move from point A to point B". Typical example is a servo moving a heavy object - constant acceleration is required.
There is another component which does not exactly belong to the architecture, but is critical to the real life applications - the behavior of the servos in case when the application controlling them dies unexpectedly - you wouldn't want your propane valve to be left open if the furnace controller segfaults, would you?
Project Implementation
First implementation was, of course, in Java. Reasons? Several:
  • It was (still is) my language of choice.
  • Best choice for rapid prototyping
  • It is really platform independent.
However, later it turned out that while Java is perfect for big boxes with lots of memory, it was not so perfect for the embedded hardware the result is supposed to be run on. This necessitated the C++ port (currently in progress, CVS module is 'servomaster-cpp'.
